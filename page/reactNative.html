ReactNative

用語

React

Web向けのライブラリ

HTML + CSS + JS で動く

主に ブラウザで動く UI を作るために使う。あくまで装飾。必須ではない。

Expo

簡易的なReactNative環境

React Native をラップして、簡単にアプリ開発できるようにしたフレームワーク

ブラウザ（Web）対応が簡単

Expo標準ライブラリがそろっている（カメラ・認証など）

React Native はネイティブ依存が大きく、環境構築やビルドでつまずくことが多かった → それを簡単にするために Expo が登場した

純 React Native

iOS/Android のネイティブコード（Swift/Kotlin）に完全自由

すべてのネイティブSDKが使える

素のプロジェクトなので細かく最適化できる

企業向け大規模アプリでは標準

外部ネイティブSDKを使う（決済・Bluetooth・独自SDKなど）

Expo、純 React Nativeのどちらを使うべきか

試しに「純 React Native」を使ってみたら初回起動が遅すぎた。api呼び出し、カメラくらいしか使わないだろうと予想してExpoを使ってみることにした。必要が出てきたら後々「純 React Native」に切り替えで良いのではないだろうか。

Android ストア（Google Play）に公開

AAB(AppBundleファイル)(.aab)形式を作成

署名用キーストアを作成。※キーストア(秘密鍵＋証明書の入れ物)


keytool -genkey -v -keystore {キーストア名}.jks -keyalg {アルゴリズム} -keysize {ビット長} -validity {有効日数} -alias {キーの別名}

{キーストア名} → my-release-key や app-release-key など、作成するファイル名
{アルゴリズム} → 通常は RSA
{ビット長} → 2048（推奨）や 4096 など
{有効日数} → 例えば 10000（約27年）署名証明書の有効期間。この日数が切れた場合、Google Play にアップロードしても アップデートとして受け付けられない。個人用テストアプリなら27年で十分。長期のアプリなら36500日(約100年)でもいい。Google Play App Signing を使えば、さらに安全に管理可能らしい。
{キーの別名} → キーストア内で識別する名前、例： my-key-alias

以下例
keytool -genkey -v -keystore torii-release-key.jks -keyalg RSA -keysize 2048 -validity 36500 -alias torii-key-alias

ルートディレクトリにあるapp.jsonファイルを修正


パッケージ名を追加
逆ドメイン形式
com.会社名.アプリ名
※Google Playに登録する場合について
Google Play に登録したパッケージ名は変更できないらしい
既に登録済みのものは使用できないらしい。またアプリごとに必ずユニーク(一意)である必要があるらしい。
同じパッケージ名のアプリは 同時にインストールできない
OSやGoogle Playはパッケージ名でアプリを管理するから

公開について

2024年から、androidアプリのアプリストアでの公開について
12人のテストユーザーの登録が必要になったらしい。
個人開発者をはじくためなのかも。
Android Studioのエミュレータでの登録という手もあるらしいが
仮に通ってもそれは正式な手順ではないらしい。
一旦ここまでとする。apkを作成までで一応アプリの作成にはなっている。
ストアで公開できるレベルの何かが出来たらまた考えよう。

ストア以外での配布について
キーストアを使った署名は必要となる。何故かというと
アップデート時にもapkをインストールすることになるが
署名の無いapkだと、更新にならずに別のアプリとして扱われるらしい。
パッケージ名の設定も不要だがやっておくとよりよい。
