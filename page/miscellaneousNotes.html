HTTPメソッド(ブラウザ)

ブラウザの制約上、実際に使用されるのは、GET、POSTの2種類らしい。

GET

データベースの書き換えが必要なく、データを取得するだけの場合に使用。

クエリパラメータでデータを受け渡す。(URL?パラメータ名1=値1&パラメータ名2=値2&)

POST

メッセージボディでデータを受け渡す。URLにパラメータがつかない。

データを変更したい時に使用。パスワードのような秘匿情報をURLに表示したくない場合。データ容量が多い場合に使用

GETだとブラウザのURLに間違って書いてしまったらそのまま実行されてしまう恐れがある。POSTの場合は、GETと違ってformを書いてあげる必要がある。誤実行を考えるとより安全と言える。

HTTPメソッド(API)

REST(REpresentational State Transfer) API「シンプルなWebシステムの設計思想」があるらしい

GET 、 POST 、 PUT 、 PATCH 、 DELETE が用意されているらしい。

同じURL(URLにリソースを書く)でHTTPメソッドだけを変えて、リソースに対する操作を行う

GET

取得

POST

リソースの作成

PUT

データが存在しているかどうかわからないときに使用します。データが存在しているときは更新をし、データが存在しない場合は新規作成を行う

PATCH

データがすでに存在しているものに対して更新をかける処理

DELETE

削除

SQL

複数のDBを更新し、整合性を保つ必要がある場合は、トランザクションを使用する。

SQLの実行回数は少ないほうが良い。SQLの実行時間が一番長いから。where条件でBulk Update(一括更新)するのが良い。IN句を使用する場合は件数の制限があるらしいので注意

【参考】SQLのIN句で指定できる値の上限について

SQLを実行するときはバインド変数を使用する。高速で実行できるのと、シングルコーテーションの入れ忘れを防ぐかららしい。

【参考】バインド変数【bind variables】ユーザー定義変数

データの件数を求める場合は、「SELECT COUNT(*) FROM テーブル名」ではなく、「SELECT COUNT(1) FROM テーブル名」を使用する。その方が早いらしい。

joinが増えれば増えるほどSQLの実行時間が長くなるらしい。joinを減らす手段の一つとして、事前にテーブルを取得してオブジェクトにid、値で持ちまわってidで取り出すのも良いらしい。

WEB・API・Batch共通

実行後、後で見たときに調査しやすいように適切にログを残す。

簡潔に、可読性・保守性の高い状態にしたい。

数字の比較について、文字型での比較と数値型での比較で結果が異なる場合があるので注意。（例　11と2の比較。文字型での比較の場合は2が大きい。左から見て1桁目で比較するためらしい。数値変換して比較することで回避できる。)

なかなか値が決まらない項目について、とりあえず"不明"という値を入れておく。そのままテストをして納品してもよい。決まった後差し替えて、その項目のみ動作確認すれば良い。

設計者と作成者が別の場合で、設計書に軽微間違い等があった場合、仕様が正しいことを確認した上で作成者が設計書を更新する。

HTTP のレスポンスステータスコード

500(Internal Server Error)

サーバー側で処理方法がわからない事態が発生したことを示します。

404(Not Found)

サーバーがリクエストされたリソースを発見できないことを示します。

403(Forbidden)

認証されていないなどの理由でクライアントにコンテンツのアクセス権がなく、サーバーが適切なレスポンスの返信を拒否していることを示します。

400(Bad Request)

「クライアントから送信されたリクエストに、何らかの原因で処理が完了できなかったこと」を示すエラー

200(OK)

リクエストが成功したことを示します。

202(Accepted)

リクエストは受理されたが、まだ実行されていないことを示します。 HTTP ではリクエストを処理した結果を示す非同期なレスポンスを後から送信する方法がないためです。

機器からリクエストを受け付け、Queueにリクエストを格納し、機器にレスポンスを返す時に202を返却していた。非同期の処理となるため。

画像形式

JPEG（Joint Photographic Experts Group）(ジェイペグ)(拡張子は「.jpg（. jpeg）」)

静止画像データのファイル形式の一種

フルカラー（約1,677万色）の画像を扱える

写真などのデータを保存するのに向いている

画質を保ったまま高圧縮ができる

上書き保存で画質が劣化する

圧縮前のデータに復元できない

PNG（Portable Network Graphics）(読み方は「ピング」もしくは「ピン」)(拡張子は「.png」)

圧縮前のデータに復元できる

透過処理ができる

比較的新しい画像ファイル形式

容量が大きくなる傾向にある。Web上で扱うときは、ページ速度に影響がでることもあるので注意

GIF（Graphics Interchange Format）読み方は「ジフ」(拡張子は「.gif」)

圧縮前のデータに復元できる

データ容量が小さい

透過処理ができる

アニメーションを作ることができる

TIFF(Tagged Image File Format)(ファイル拡張子は.tiffまたは.tif)

メリット:画像データを圧縮せずに保存することで画像が劣化しない

メリット:JPEGよりもきれいに印刷できるという点で、印刷物などに広く使われている

デメリット:無圧縮のためファイルサイズが他の拡張子と比べて格段に大きい

写真家に支持されている

BMP（ビットマップ/ビーエムピー）(Microsoft Windows Bitmap Imageの略)( 拡張子「.bmp」)

メリット:画像を圧縮せずに保存できる。

メリット:解像度の高い画像が必要な場合に使える。

デメリット:ファイルサイズが大きいこと

デメリット:Webでは扱えないこと
